# AI Usage Disclosure

## 1. Project Requirement Prompt (PRP) used

The complete brief, architecture, and acceptance criteria given to the AI agent is `doc/PRP.md` in this repository. It expands the original 60-minute take-home brief (`doc/Assessment-Readme.md`) into an 8-phase, gated spec (Phase 0–7) with a Requirements Traceability Matrix (§8) and a Definition of Done (§9). Before writing any code, the PRP was read in full, cross-checked against `doc/Assessment-Readme.md` for contradictions/gaps, and the resulting review (architecture summary, phase-by-phase plan, and ambiguity call-outs) was presented to the human for sign-off before implementation began.

## 2. Review mechanism

A layered gate was used throughout, walked after every phase rather than only at the end:

- **(a) TypeScript `strict: true`** as the first reviewer — added explicitly to `tsconfig.app.json` since the default Vite scaffold didn't set it. No `any` anywhere in `src/` (verified by grep); no non-null `!` is used anywhere in the codebase (the PRP's one sanctioned `!` in the mock API was removed in favor of a safe fallback — see `doc/NOTES.md`).
- **(b) ESLint** with `eslint-plugin-react-hooks`'s flat-recommended config, including the newer `react-hooks/set-state-in-effect` and `react-hooks/refs` rules, both of which caught real bugs during development (a synchronous `setState` in an effect, and a ref read during render) — see below.
- **(c) Vitest**: 70 tests across 11 files — `filterTasks`/`sortTasks`/`urlState` (pure logic), `devFlags`/`useDebouncedValue` (small focused hooks/utils), `useTasks`/`mockApi` (loading→success/error, refetch, optimistic update + rollback, cache-on-remount, the abort path, and CRUD: create/update/delete), `FilterContext` (reducer behavior + a regression test for the re-render-loop fix below), `VirtualList` (windowing math, the virtualization badge), `TaskFilters` (facet counts, toggling, the debounced search), and `ErrorBoundary` (catch + reset recovery).
- **(d) The Requirements Traceability Matrix (§8)** walked row by row against the implementation.
- **(e) `npm run build`** as the production gate — also used to confirm `TaskStats`, `TaskModal`/`ConfirmDialog`/`Snackbar`, and the 1,000-row dataset (`mockData_1k.json`, dynamically imported) each emit as separate chunks, so none bloats the default bundle.
- **(f) A scripted manual smoke test** via a headless-Chromium Playwright driver, run against the live dev server rather than just inspecting code: initial load, status/priority filter toggling, debounced assignee search, optimistic status change, over-constrained filters producing the empty state, the `?throwRender=1` flag triggering the `ErrorBoundary` fallback, the `?failTasks=1` flag triggering optimistic-update rollback, and the `?syntheticTasks=1` large-dataset toggle. This caught real bugs that `typecheck`/`lint`/`build` did not: an infinite re-render loop in `FilterContext` (the context value and its dispatcher functions were recreated on every render, so the debounced-search effect re-fired forever), a `VirtualList` row-height mismatch that caused task cards to visually overlap once icons/badges were added, a later recurrence of that overlap via an unbounded `TaskCard` description (fixed with `line-clamp-2` + a `max-h` backstop — see `doc/NOTES.md`), and — while writing tests — a missing RTL `cleanup()` between tests plus two cases of `vi.advanceTimersByTime` needing an `act()` wrapper to flush the resulting state update.

## 3. Justification of the implemented solution

Vite was chosen over Create React App (CRA) because CRA is officially deprecated and the brief explicitly permits any scaffolder (a `start` script alias to `vite` is kept so the brief's literal `npm start` instruction still works). Filter state lives in React Context + `useReducer` because that's the literal "Context" requirement at a scale (a few thousand rows) where heavier state management would be unjustified. `useTasks` is hand-written because the brief requires custom-hook primitives (cache, abort, optimistic update, rollback) rather than a data-fetching library. `VirtualList` is custom rather than `react-window`/`react-virtual` so the windowing mechanism itself is demonstrated, with a code comment explaining the harder variable-height strategy that real 10,000+ row data with non-uniform card heights would need (a measured-height cache + cumulative offsets + `ResizeObserver`, as those libraries implement internally); it also shows a "Rendering N of M (virtualized)" badge as visible proof of windowing, since a virtualized list otherwise looks identical to a fully-rendered one. `ErrorBoundary` is a class component because hooks cannot catch render errors thrown by descendants — only `getDerivedStateFromError`/`componentDidCatch` can. Filtering happens client-side with a debounced assignee search because the dataset is small enough that server-side filtering would add latency for no benefit, while the debounce keeps the search input from dispatching (and re-rendering the whole filtered tree) on every keystroke.

Several later, explicitly user-directed changes go beyond the original PRP, each with its own rationale fully logged in `doc/NOTES.md`:

- A **Tailwind CSS redesign** against a supplied visual mockup (superseding the PRP's original "no Tailwind" choice), adding a header bar, filter facet counts, a sort control, and a hand-built SVG donut-chart stats panel.
- **Mock fixture data moved** out of `mockApi.ts` into `src/api/data/{mockData.json, mockData_1k.json}` — the latter loaded via a dynamic `import()` so the 1,000-row dataset used to exercise `VirtualList` never ships in the default production bundle.
- **Filter and sort state synced to the URL** (`src/utils/urlState.ts`) via `URLSearchParams` + `history.replaceState`, so reloading or sharing a link preserves them, without pulling in a router for two small, already-serializable pieces of state.
- **A full create/edit/delete task flow**: a "Create Task" button in `Header`, a pencil icon on each `TaskCard` opening the same modal pre-filled for editing, a confirmation dialog gating delete, and a snackbar confirming each action. `mockApi.ts` gained a real mutable in-memory store for these three operations, deliberately kept separate from the original `updateTaskStatus`'s non-mutating, rollback-on-failure demo contract — the new operations are real CRUD expected to persist for the session (including surviving a `refetch`), not a failure-injection demo.
- **`TaskActionsContext` split into two contexts** (data vs. action-triggers) rather than one, specifically so that `Header`/`TaskCard` consuming the stable action triggers doesn't break `React.memo` on `TaskCard` — a single combined context would re-render every card on any task mutation regardless of memo, since the context's own value reference would change. Verified live via the `?renderCounter=1` flag that unrelated cards still don't re-render after this change.
- **`TaskModal`/`ConfirmDialog`/`Snackbar` code-split via `React.lazy`**, the same pattern as `TaskStats`, since none of the three is needed until a user opens a modal/dialog/toast — confirmed via `npm run build` emitting each as its own chunk.
