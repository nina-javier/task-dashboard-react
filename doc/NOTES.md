# Notes & trade-offs

- **Vite over CRA**: CRA is deprecated; Vite gives faster dev/build with native ESM and no extra config for the React 19 + TS strict combination this assessment exercises.
- **Custom `useTasks` hook** (cache + `AbortController` + optimistic update/rollback) instead of a data-fetching library: the brief explicitly requires hand-built custom-hook primitives, and the dataset/app are small enough that a library would be unjustified weight.
- **Custom `VirtualList`** (fixed-row windowing) instead of `react-window`/`react-virtual`: demonstrates the windowing mechanism directly per the brief; a documented comment in `VirtualList.tsx` explains the harder variable-height strategy needed for 10,000+ rows, which isn't implemented since this dataset's cards are uniform height.
- **Class `ErrorBoundary`**: hooks cannot catch render errors thrown by descendants, only `getDerivedStateFromError`/`componentDidCatch` can â€” hence the one class component in an otherwise hooks-only codebase.
- **Client-side filtering + debounced assignee search** (`useDebouncedValue`, 250ms): the mock dataset is small, so filtering in-memory keeps interactions instant; debouncing only protects the search input from firing a dispatch (and therefore a context re-render) on every keystroke.
- **Single `src/devFlags.ts`** module (URL-query-param based) backs all three "prove it" demo mechanisms â€” the failure injector, the render-error throw, and the render counter â€” instead of three bespoke ad hoc toggles, so there's one documented surface instead of three.
- **`doc/` not `docs/`**: the PRP text uses both spellings inconsistently; the repo's actual directory is singular `doc/` (already containing `PRP.md`, `Assessment-Readme.md`), so all `docs/...` references in the PRP are treated as that.
- **Mock API's `find(...)` no longer uses a bare `!`**: the PRP's embedded mock asserts non-null on `TASKS.find(...)`, which is fine for the original 5-row dataset but unsafe against the larger dataset (below). `updateTaskStatus` now falls back to a constructed task instead of asserting â€” the non-null-assertion exception is therefore not used anywhere in this codebase.
- **Mock data moved to `src/api/data/{mockData.json, mockData_1k.json}`**, imported into `mockApi.ts` rather than inlined: keeps the fixture data out of the API logic, and `resolveJsonModule: true` was added to `tsconfig.app.json` so TS can type the imports. `mockData_1k.json` is a deterministic, generated 1,000-row dataset (same shape as `mockData.json`) purely for exercising `VirtualList`'s windowing â€” it isn't meant to look hand-authored.
- **Dev-only large-dataset toggle** (`?syntheticTasks=1`) swaps in `mockData_1k.json` so `VirtualList`'s windowing is exercised against a large list, not just trusted by inspection â€” the 5-row mock alone never engages the windowing branch. `mockData_1k.json` is loaded via a dynamic `import()` (cached after first load) rather than a static import, so its ~180KB never bloats the production bundle for an app that ships the 5-row dataset by default â€” confirmed via `npm run build` emitting it as its own chunk.
- **Failure injector treated as required, not optional**: although PRP Â§3 calls it an "optional enhancement," Phases 2/5/9 and the traceability matrix all depend on it to demonstrate rollback, so it was implemented as load-bearing.
- **Tailwind CSS added post-hoc, superseding PRP Â§1's "no Tailwind" choice**: the user explicitly directed a Tailwind-based redesign against a supplied mockup after the original phases were complete. Plain CSS (`App.css`/custom classes) was removed and replaced with Tailwind utility classes; this is a deliberate, explicit override of the PRP's original styling decision, not a contradiction introduced by the agent.
- **Hand-built SVG donut chart** (`DonutChart.tsx`) instead of a charting library: the mockup's "Task Overview" panel needed one, and a single conic-style stroke-dasharray `<svg>` component is small enough not to justify a dependency.
- **Client-side sort** (`sortTasks.ts`, newest/oldest by `createdAt`) added to match the mockup's "Sort by" control â€” not in the original PRP scope, implemented as local `Dashboard` state applied after filtering, kept as a pure function alongside `filterTasks` for the same testability reason.
- **Assignee filter UX changed from free-text-only to search-narrowed radio list**: the mockup shows an explicit single-select (`All`/name radios) plus a search box that narrows which names are shown, rather than the search box directly driving the task filter. This still satisfies the PRP's "single-select or search" requirement and keeps `useDebouncedValue` in the loop (debouncing the list-narrowing), but the filtering itself is now driven by the radio selection.
- **`VirtualList` shows a "Rendering N of M (virtualized)" badge** whenever windowing is actually doing work (`items.length > visibleItems.length`). Windowing is invisible by design â€” a virtualized list looks and scrolls identically to a fully-rendered one â€” so without this, the only way to prove it's not just rendering everything behind a scrollbar is to open DevTools and count `.task-card` nodes. The badge makes that provable in the UI itself; it's not gated behind a query-param flag because it only ever appears once there's something to prove (small lists never trigger it).
- **Test coverage expanded to 16 tests, prioritized by regression risk over breadth**: added `FilterContext.test.tsx` (toggle/clear behavior, plus a regression test pinning the `setAssignee`-same-value no-op that fixed the earlier infinite-render-loop bug) and extended `useTasks.test.tsx` with the error path, `refetch`, and the cache-on-remount behavior. Still untested: `sortTasks.ts`, `VirtualList`'s windowing math, `ErrorBoundary`'s catch/reset, and every component's actual rendered output (no RTL render tests exist yet â€” coverage stops at hooks and pure functions).
- **`src/test/setup.ts` was missing RTL's `cleanup()` between tests** â€” discovered while adding `ErrorBoundary.test.tsx`: without it, each test's rendered DOM accumulated across the file instead of unmounting, so a later test's queries could silently match a previous test's leftover markup. Fixed globally in `setup.ts` (`afterEach(cleanup)`) rather than per-file, since every future RTL test would otherwise hit the same gap.
- **Testing a "throw once, then recover" child is not a reliable React 19 pattern**: React's concurrent renderer retries a failed render synchronously before surfacing the error, so a component that flips a module-level flag on its first throw (`if (!hasThrown) { hasThrown = true; throw â€¦ }`) sees that flag already set by the retry and silently stops throwing â€” the boundary never reliably observes the error. `ErrorBoundary.test.tsx` instead controls throwing via a `shouldThrow` prop and uses RTL's `rerender` to change it, then asserts the boundary's error state persists across that rerender (matching real `getDerivedStateFromError` semantics) until `reset()` is explicitly called.
- **Added `devFlags.test.ts` and `mockApi.test.ts`**, raising coverage to 36 tests: `devFlags` covers each flag's default-off state and exact-match-on-"1" behavior; `mockApi` covers what `useTasks.test.tsx` mocks away â€” the fetch/update timing, the failure-injector and large-dataset branches, the abort path, the unknown-id fallback, and that the source dataset is never mutated. `mockApi.test.ts` mixes fake timers (for the 300ms/800ms delays) and real timers (for the large-dataset case, since loading `mockData_1k.json` is a real dynamic import that fake timers don't drive â€” mixing the two caused a real test hang during development).
- **Added `useDebouncedValue.test.ts`, `VirtualList.test.tsx`, and `TaskFilters.test.tsx`**, raising coverage to 51 tests. `VirtualList` needed a `data-testid="virtual-list-viewport"` added to its scroll container purely for test targeting (no behavior change) so a `scroll` event could be fired on it directly.
- **`vi.advanceTimersByTime` must be wrapped in `act()` when it triggers a React state update** (e.g. `useDebouncedValue`'s `setTimeout` callback calling `setDebounced`): without `act()`, advancing the fake clock runs the timer callback, but React doesn't flush the resulting re-render before the next assertion reads `result.current`/queries the DOM, so the test observes the pre-update value and fails even though the hook is correct. This bit both `useDebouncedValue.test.ts` and `TaskFilters.test.tsx`'s debounce test; the fix is `act(() => vi.advanceTimersByTime(ms))` instead of a bare call.
- **Added `"start": "vite"` to `package.json`**: the original assessment brief's Submission section literally says "the app must compile and run with `npm start`," a CRA-era convention; Vite uses `dev`. The alias satisfies that literal instruction without changing tooling.
- **Removed the header's hamburger icon + user/avatar control and `TaskCard`'s "..." menu**, all at explicit request. None of these were ever required by the PRP or the original brief â€” they were decorative affordances carried over from the supplied visual mockup, with no wired-up behavior (no nav, no auth, no menu actions). Removing them drops unused surface, not a requirement; `doc/PRP.md`'s component hierarchy (`Appâ†’Dashboardâ†’{TaskFilters, TaskListâ†’TaskCard, TaskStats}`) and traceability matrix are unaffected since `Header` was never part of either.
- **Filter/sort URL persistence** (`src/utils/urlState.ts`), closing the "Trade-offs / what's next" item from the original README: filters (`FilterContext`) and sort order (`Dashboard`'s local state) each lazily read their initial value from `window.location.search` on mount and re-serialize on every change via `history.replaceState`. No router was added â€” `URLSearchParams` plus a read-merge-write helper (`replaceUrl`) is enough for two independent pieces of state to share a query string without clobbering each other or any other param. `replaceState` (not `pushState`) was chosen so toggling a checkbox doesn't pollute browser back/forward history with one entry per click. Status/priority are comma-joined lists; assignee is the raw string; sort and empty/default values are omitted from the URL entirely to keep it clean (`?status=todo,done&assignee=Bob` rather than `?status=todo,done&priority=&assignee=Bob&sort=newest`). Covered by 11 new tests in `urlState.test.ts` (parsing, invalid-value filtering, round-tripping, default-omission, preserving unrelated params); `src/test/setup.ts` now resets `window.location` after every test so this doesn't leak state across test files.
- **Fixed a real bug in `FilterContext`'s `useReducer` call** while picking up the URL-persistence work above: `useReducer(reducer, initialFilters, readFiltersFromUrl)` passes `initialFilters` (a `Filters` object) as the argument *to* `readFiltersFromUrl`, not as a fallback â€” so `readFiltersFromUrl` received a `Filters` object where it expected a URL search string, and `new URLSearchParams(thatObject)` silently coerced `assignee: null` into the string `"null"`. Fixed by wrapping the initializer (`() => readFiltersFromUrl()`) so it ignores the arg and always reads the real `window.location.search`.
- **Full create/edit/delete task flow**, added on top of the original status-only optimistic update: a "Create Task" button in `Header`, a pencil icon on each `TaskCard` opening the same `TaskModal` pre-filled for editing, a `ConfirmDialog` gating delete, and a `Snackbar` confirming create/save/delete. `mockApi.ts` gained a real *mutable* in-memory `store` (seeded via `structuredClone(mockData)`) with `createTask`/`updateTask`/`deleteTask`, deliberately separate from `updateTaskStatus`'s existing non-mutating contract â€” the inline status `<select>` stays a self-contained demo of optimistic update + rollback (already tested, still doesn't touch `store`), while the new modal-driven mutations are real CRUD that's expected to survive a `refetch`. create/update/delete are awaited directly (no optimistic-apply-then-rollback) rather than reusing the optimistic pattern, since they're modal-gated (the UI already shows a pending state via a disabled Save/Delete button) and the mock never rejects them.
- **Split `TaskActionsContext` into two contexts** (`TaskDataContext` for `{state, refetch, setStatus}`, `TaskActionsContext` for `{openCreate, openEdit}`) rather than one: `Header` and `TaskCard` only need the latter to trigger the create/edit modal, and that value's `useCallback`s never change reference, so subscribing to it doesn't break `React.memo` on `TaskCard`. A single combined context would have made every card re-render on *any* task mutation (since `state` changes by reference on every create/update/delete/status-change), regardless of memo â€” verified live via `?renderCounter=1`: toggling a filter or editing/creating/deleting an unrelated task produces zero render logs for untouched cards, while the one card whose own data actually changed still re-renders as expected.
- **`Header`/`Dashboard` can't share `useTasks()` by calling it twice** â€” each call creates an independent `useState`, so a mutation applied via one instance wouldn't be visible to the other despite sharing the same module-level cache for the *initial* read. `useTasks()` is called exactly once, inside `TaskActionsProvider`, and both data and actions are threaded out via the two contexts above.
- **`TaskModal`/`ConfirmDialog`/`Snackbar` lazy-split via `React.lazy`/`Suspense`**, same pattern as `TaskStats`: they're imported in `TaskActionsContext.tsx` (where they're rendered) rather than eagerly, since none of the three is needed until a user actually opens a modal/dialog/toast. `npm run build` confirms each now emits as its own chunk (`TaskModal-*.js`, `ConfirmDialog-*.js`, `Snackbar-*.js`, plus their shared `Modal.tsx` dependency), and the main bundle dropped from 222 kB â†’ 216 kB raw (68.9 kB â†’ 67.9 kB gzip). `fallback={null}` is fine here since these are tiny chunks behind a user-initiated click, not the initial page load.
- **Fixed a text-overlap bug in `TaskCard`**: `VirtualList` absolutely-positions every row at a fixed `ROW_HEIGHT` (140px, set in `TaskList.tsx`), but `TaskCard`'s description had no line limit â€” a long enough description grew taller than 140px and visually bled into the next absolutely-positioned row, making two adjacent cards look like one merged card. Fixed with two layers: `line-clamp-2` on the description (bounds it to 2 lines in the common case) and `max-h-[132px] overflow-hidden` on the card itself (a hard backstop slightly under the 140px row height, so it can never overlap the next row even in an extreme case neither `line-clamp` nor normal content would produce). The `132px` is coupled to `TaskList.tsx`'s `ROW_HEIGHT` constant by convention, not by a shared import â€” if `ROW_HEIGHT` changes, this needs a matching update. Verified by launching the dev server and using a scripted Playwright flow (`Create Task` â†’ long description â†’ screenshot): the long card now clips cleanly within its own border and the next card renders without overlap, with no console errors. No automated regression test exists yet for this (`TaskCard` itself has no test file â€” see `README.md`'s Testing section).
